# uidesign PRD 产品需求文档

## 1. 文档信息

- 项目名称：uidesign
- 文档类型：PRD 产品需求文档
- 当前版本：V0.1
- 编写日期：2026-05-27
- 产品定位：面向设计师、前端开发和验收人员的设计还原度自动走查工具

## 2. 项目背景

在前端页面开发完成后，设计师、产品经理或验收人员通常需要人工对比设计稿和开发页面，检查页面是否按设计稿还原。这个过程耗时、重复、容易漏看，尤其是在页面较多、组件复杂、迭代频繁时，人力成本很高。

当前团队面临的核心问题是：开发出来的页面与设计稿的还原度不够高，但人工走查效率低，现有自动化检测精度也不理想。需要一个工具自动对比设计稿与开发页面，发现视觉、文本、布局、颜色、间距等差异，并生成清晰的问题汇总，帮助设计和研发团队快速定位并修复问题。

## 3. 产品目标

### 3.1 业务目标

- 降低设计还原度走查的人力成本。
- 提升设计稿与开发页面的对比效率。
- 减少人工验收遗漏的问题。
- 形成结构化走查报告，方便设计师、开发和产品协同修复。
- 建立可持续优化的还原度检测精度体系。

### 3.2 产品目标

- 支持上传设计稿截图和开发页面截图，自动识别两者差异。
- 支持对差异进行分类、定位、严重程度标记和问题汇总。
- 支持输出面向人的问题说明，而不是只输出算法结果。
- 支持用户复核误判结果，为后续精度优化提供反馈数据。
- 第一阶段优先解决截图级对比，不直接扩展到完整 Figma 文件解析或代码级分析。
- 第一阶段仅支持手动上传截图，不支持输入 URL 自动截图。
- 第一阶段不做用户账号、历史记录和团队空间，只做本地单次走查任务。

## 4. 目标用户

### 4.1 设计师

关注页面是否符合设计稿，重点看视觉一致性、间距、颜色、字体、组件状态和图标图片是否正确。

### 4.2 前端开发

关注具体哪里不一致、应该改什么、差异位置在哪里，以及修复优先级。

### 4.3 产品经理 / QA / 验收人员

关注页面是否达到上线验收标准，问题是否被完整记录、跟踪和关闭。

## 5. 核心使用场景

### 5.1 单页走查

用户上传一张设计稿截图和一张开发页面截图，系统自动对比并输出差异结果。

典型流程：

1. 用户上传设计稿截图。
2. 用户上传开发页面截图。
3. 系统自动进行图像预处理和尺寸对齐。
4. 系统识别布局、文本、颜色、间距、图片图标等差异。
5. 系统在截图上标记问题区域。
6. 系统生成问题列表和还原度评分。
7. 用户复核问题，标记正确、误判或忽略。
8. 用户导出走查报告。

### 5.2 多页面批量走查

用户一次上传多组设计稿和开发截图，系统批量对比并生成汇总报告。

第一版可作为次优先级功能，先保证单页走查精度和体验。

### 5.3 当前任务报告复核

用户在当前单次走查任务中查看检测结果，筛选问题类型、严重程度和复核状态，确认哪些问题需要修复，哪些属于误判或可忽略问题。

第一版不做账号体系、历史记录和跨任务追踪。用户刷新页面或结束任务后，系统不保证保留历史数据。完整问题流转可放到后续版本。

### 5.4 Web 与 App 页面走查

第一版同时支持 Web 端页面截图和移动端 App 页面截图。系统只关心输入图片本身，不要求知道页面来自浏览器、H5、小程序、iOS App 或 Android App。

需要注意的是，Web 和 App 的截图比例、分辨率、字体渲染方式可能不同，系统需要在报告中记录图片尺寸，并在对齐失败或疑似非同一状态时提示用户复核。

## 6. 产品范围

### 6.1 MVP 范围

MVP 优先支持以下能力：

- 上传设计稿截图和开发页面截图。
- 支持 Web 端截图和移动端 App 截图。
- 自动对齐两张截图的尺寸和主要内容区域。
- 使用视觉检测识别像素、布局、颜色、位置、大小等差异。
- 使用 OCR 识别截图中的文本内容和文本位置。
- 使用 GPT-4o 对结构化检测结果进行解释和归纳，生成自然语言问题说明。
- 输出差异标注图、问题列表、问题分类、严重程度和还原度评分。
- 支持用户标记检测结果为正确、误判、忽略。
- 支持导出 Markdown 和 PDF 走查报告。

### 6.2 暂不纳入 MVP

- 直接解析 Figma 源文件。
- 直接连接浏览器自动截图。
- 输入 URL 自动截图。
- 用户账号、登录注册、历史记录。
- 代码级自动修复。
- 与 Jira、飞书、企业微信等平台深度集成。
- 复杂响应式多断点全自动走查。
- 完整团队权限、审批流和任务流转系统。

这些能力可以作为后续版本规划。

## 7. 核心功能需求

### 7.1 文件上传

用户可以上传设计稿截图和开发页面截图。

需求说明：

- 支持 PNG、JPG、JPEG 格式。
- 单张图片大小建议限制在 20MB 以内。
- 上传后展示图片预览。
- 明确区分设计稿截图和开发页面截图。
- 支持上传 Web 端截图和移动端 App 截图。
- 第一版不提供 URL 输入框，不自动打开页面截图。
- 上传失败时给出清晰原因，例如格式不支持、文件过大、图片损坏。

验收标准：

- 用户能成功上传两张截图并进入对比流程。
- 图片预览清晰，不能出现明显压缩失真影响判断。
- 系统能识别缺失文件并阻止开始对比。

### 7.2 图片预处理与对齐

系统需要在对比前处理两张图片，减少因尺寸、裁切、缩放导致的误判。

需求说明：

- 自动识别两张图的尺寸差异。
- 支持按比例缩放到统一对比尺寸。
- 尝试识别主要内容区域，减少空白区域干扰。
- 记录预处理方式，便于报告中说明。
- 当系统无法可靠对齐时，提示用户手动确认或重新上传。

验收标准：

- 同一页面在轻微尺寸差异下仍能正常对比。
- 因尺寸不一致导致的大面积误判应明显减少。
- 对齐失败时不能给出看似确定但明显错误的报告。

### 7.3 视觉差异检测

系统通过 OpenCV 对两张截图进行视觉对比，发现肉眼容易漏掉的差异。

检测类型：

- 颜色差异：背景色、文字色、按钮色、边框色等不一致。
- 位置差异：元素整体偏移、对齐错误。
- 尺寸差异：元素宽高、图标大小、图片尺寸不一致。
- 间距差异：内边距、外边距、组件间距不一致。
- 结构差异：模块缺失、多出模块、组件顺序错误。
- 图形差异：图片、图标、按钮形态与设计稿不一致。

验收标准：

- 系统能在截图上框选主要差异区域。
- 系统能将视觉差异归类到合理的问题类型。
- 对大面积背景、阴影、抗锯齿造成的微小差异，需要降低误报。

### 7.4 文本识别与文本差异检测

系统通过 PaddleOCR 识别截图中的文本内容、位置和文本框范围。

检测类型：

- 文案错误。
- 文本缺失。
- 多出文本。
- 文本位置偏移。
- 字号、字重、行高疑似不一致。
- 换行位置异常。

需求说明：

- OCR 结果需要包含文本内容、置信度和坐标。
- 文本对比需要支持相似文本匹配，避免轻微 OCR 误识别导致大量误报。
- 对低置信度文本需要标记为“需人工确认”。

验收标准：

- 常规中文、英文、数字文本能被识别并参与对比。
- 文本内容差异能明确展示设计稿文本和开发页面文本。
- OCR 置信度较低时，系统不能给出过度确定的结论。

### 7.5 差异理解与问题生成

系统使用 GPT-4o 对 OpenCV 和 PaddleOCR 的检测结果进行理解、归纳和表达。

需求说明：

- 将算法检测结果转化为设计师和开发能理解的问题描述。
- 合并同一区域内的重复问题，避免报告过长。
- 根据差异类型和影响范围给出严重程度建议。
- 输出建议修复方向，例如“按钮高度偏小”“标题文字缺失”“卡片间距比设计稿大”。
- GPT-4o 只作为解释和辅助判断，不作为唯一检测来源。

验收标准：

- 问题描述清楚、可执行，开发能知道应该检查哪里。
- 同一区域重复问题应尽量合并。
- 对不确定问题需要明确标记“疑似”或“需人工确认”。

### 7.6 差异标注图

系统需要在图片上可视化展示差异区域。

需求说明：

- 支持在开发页面截图上标记差异区域。
- 支持点击问题列表定位到对应区域。
- 差异标记需要醒目但不遮挡内容。
- 不同问题类型可以使用不同颜色或标签。

验收标准：

- 用户能快速从列表定位到截图中的问题区域。
- 标注不会严重遮挡原图内容。
- 标注区域与实际差异位置基本一致。

### 7.7 问题列表

系统需要以结构化方式展示所有检测到的问题。

字段建议：

- 问题编号。
- 问题类型。
- 严重程度。
- 所在区域。
- 问题描述。
- 设计稿表现。
- 开发页面表现。
- 建议修复方向。
- 置信度。
- 用户复核状态。

问题类型建议：

- 文本问题。
- 颜色问题。
- 间距问题。
- 对齐问题。
- 尺寸问题。
- 图片 / 图标问题。
- 组件缺失。
- 多余元素。
- 结构布局问题。
- 需人工确认。

严重程度建议：

- 高：影响主要内容、核心操作或明显视觉错误。
- 中：影响局部视觉一致性，但不阻碍使用。
- 低：轻微偏差或需要人工确认的细节问题。

验收标准：

- 用户能按问题类型、严重程度、复核状态筛选。
- 每个问题都能关联到截图中的位置。
- 问题描述不能只有技术术语，需要可读、可执行。

### 7.8 还原度评分

系统需要给出页面整体还原度评分，帮助用户快速判断页面质量。

评分建议：

- 总分：0-100 分。
- 按问题类型扣分。
- 高严重问题扣分更多。
- 低置信度问题扣分较低或不计入最终分，只作为参考。

评分维度：

- 文本一致性。
- 颜色一致性。
- 布局一致性。
- 间距一致性。
- 组件完整性。
- 图片图标一致性。

验收标准：

- 报告中展示总分和各维度得分。
- 分数需要能解释，不能只是给一个数字。
- 用户能看到主要扣分原因。

### 7.9 人工复核反馈

系统需要允许用户标记检测结果是否准确，用于减少误判并积累优化数据。

复核状态：

- 正确。
- 误判。
- 忽略。
- 需后续确认。

需求说明：

- 用户可以对单个问题进行复核。
- 用户可以添加备注。
- 系统记录用户反馈，为后续优化检测阈值和提示词提供依据。

验收标准：

- 用户能修改问题复核状态。
- 报告中能区分系统识别问题和人工确认问题。
- 误判数据可以被统计。

### 7.10 报告导出

系统需要支持导出走查结果，方便团队共享和修复。

导出内容：

- 页面基本信息。
- 上传图片信息。
- 还原度总分。
- 各维度得分。
- 差异标注图。
- 问题列表。
- 用户复核状态。
- 生成时间。

导出格式：

- MVP 同时支持 Markdown 和 PDF。
- 后续可支持 Excel、JSON、在线分享链接。

验收标准：

- 导出的报告内容完整。
- 报告能被开发和设计直接阅读。
- 报告中的问题编号与系统页面一致。

## 8. 精度要求与评估指标

### 8.1 检测精度目标

第一阶段不追求完全替代人工，而是减少人工初筛成本。系统应优先做到“主要问题不漏掉，误判可以被复核”。

建议目标：

- 主要视觉差异召回率：不低于 80%。
- 文本内容差异召回率：不低于 90%。
- 严重问题召回率：不低于 90%。
- 有效问题准确率：目标达到 90% 左右，即系统输出的问题中，大部分应能被用户确认为真实问题。
- 问题误报率：第一版需要控制在可人工复核范围内，不能用大量低价值问题淹没有效问题。
- 单页对比耗时：普通页面建议控制在 30 秒以内。

### 8.2 精度评估方式

需要建立测试样本集：

- 收集真实设计稿截图和开发页面截图。
- 人工标注标准答案，包括问题类型、位置和严重程度。
- 使用标准答案对系统检测结果进行评估。
- 定期统计召回率、准确率、误报率、漏报率。

核心指标：

- 召回率：真实存在的问题中，系统发现了多少。
- 准确率：系统发现的问题中，有多少是真问题。
- 定位准确率：系统标记的位置是否接近真实问题区域。
- 分类准确率：系统是否把问题归到正确类型。
- 文案可用率：生成的问题描述是否能指导开发修复。

### 8.3 可能误判场景

- 截图尺寸不一致。
- 页面滚动位置不一致。
- 字体渲染抗锯齿差异。
- 阴影、渐变、透明度导致的轻微像素变化。
- OCR 把相似字符识别错误。
- 动态内容、时间、头像、数据占位不同。
- 设计稿与开发页面本身不是同一状态。

系统需要对这些场景给出提示，并尽量通过预处理、阈值控制和人工复核降低影响。

## 9. 技术方案方向

本 PRD 不限定具体实现细节，但建议采用“机器检测 + OCR + 大模型解释”的组合方式。

### 9.1 OpenCV

OpenCV 用于视觉对比，主要负责发现颜色、位置、尺寸、边缘、区域和布局差异。它适合做稳定、可重复的图像检测，是还原度走查的基础检测层。

### 9.2 PaddleOCR

PaddleOCR 用于识别截图中的文字内容和文字位置，主要解决文案是否正确、是否缺失、是否偏移、换行是否异常等问题。

### 9.3 GPT-4o

GPT-4o 用于理解和组织检测结果，生成更接近人工走查语言的问题描述。它不应单独决定所有差异，而是结合 OpenCV 和 PaddleOCR 的结构化结果做解释、合并和总结。出于数据安全考虑，第一版默认只向 GPT-4o 发送结构化检测结果，不直接发送原始截图。

### 9.4 推荐处理流程

1. 图片上传。
2. 图片预处理和对齐。
3. OpenCV 进行视觉差异检测。
4. PaddleOCR 进行文本识别和文本对比。
5. 差异结果结构化。
6. GPT-4o 对结构化结果进行解释和归纳，默认不接收原始截图。
7. 生成标注图、问题列表和还原度评分。
8. 用户复核并导出报告。

## 10. 非功能需求

### 10.1 性能

- 单页普通截图对比建议控制在 30 秒以内。
- 大图处理时需要显示进度或等待状态。
- 批量任务需要避免阻塞单页查看。

### 10.2 稳定性

- 上传失败、OCR 失败、GPT-4o 调用失败时需要给出明确提示。
- 外部服务超时后应支持重试。
- 报告生成失败时不能丢失已完成的检测结果。

### 10.3 安全与隐私

- 截图可能包含未公开产品信息，需要谨慎处理。
- 不应在代码或文档中写死 API Key。
- 调用外部模型服务前应明确数据发送范围。第一版默认只发送结构化检测结果给 GPT-4o，不发送原始截图。
- 后续如支持团队使用，需要补充数据保存周期和删除机制。

### 10.4 可解释性

- 系统需要说明为什么判断为问题。
- 对低置信度问题需要提示用户复核。
- 还原度评分需要展示扣分原因。

## 11. 页面与交互需求

### 11.1 单次任务入口页

核心内容：

- 新建走查任务入口。
- 当前任务状态。
- 不展示最近走查记录和历史报告入口。

### 11.2 新建走查任务页

核心内容：

- 上传设计稿截图。
- 上传开发页面截图。
- 图片预览。
- 开始对比按钮。
- 上传要求和错误提示。

### 11.3 对比结果页

核心内容：

- 设计稿截图。
- 开发页面截图。
- 差异标注视图。
- 问题列表。
- 筛选和排序。
- 还原度评分。
- 导出报告按钮。

交互要求：

- 点击问题列表可以定位截图区域。
- 点击截图标注可以高亮对应问题。
- 用户可以修改问题状态。

## 14. 风险与应对

### 14.1 检测精度不稳定

风险：不同页面、不同截图质量下检测效果差异较大。

应对：建立标准测试集，持续统计召回率和误报率，先覆盖常见页面类型。

### 14.2 误报过多

风险：用户需要花大量时间排除误判，影响信任。

应对：引入置信度、严重程度和人工复核机制，优先展示高置信度和高严重问题。

### 14.3 GPT-4o 输出不稳定

风险：大模型可能描述不一致或过度推断。

应对：将输入限制为结构化检测结果，要求输出固定格式，并对不确定问题标记“疑似”。

### 14.4 OCR 识别错误

风险：文字识别错误导致文本差异误判。

应对：使用 OCR 置信度、相似度匹配和人工复核降低影响。

### 14.5 数据安全风险

风险：设计稿和页面截图可能包含敏感信息。

应对：明确数据处理范围，避免默认上传第三方服务，必要时提供本地处理或脱敏方案。

## 15. 已确认产品约束

- 第一版只支持手动上传截图，不支持输入 URL 自动截图。
- 第一版报告同时支持 Markdown 和 PDF 导出。
- 第一版不做用户账号、历史记录和团队空间，只做本地单次走查任务。
- 第一版同时支持 Web 端截图和移动端 App 截图。
- 第一版精度目标以 90% 左右的有效问题准确率作为阶段目标，后续通过人工复核数据持续优化。
- 出于安全考虑，第一版默认只向 GPT-4o 发送结构化检测结果，不直接发送原始截图。

## 16. 产品结论

uidesign 的第一阶段应聚焦在“截图级设计还原度自动走查”。产品核心不是简单比较两张图片，而是帮助用户高效发现开发页面与设计稿之间的可修复问题。

第一版应优先做稳单页流程：手动上传截图、自动检测、问题标注、问题列表、还原度评分、人工复核和 Markdown / PDF 报告导出。技术上建议用 OpenCV 做视觉检测，PaddleOCR 做文本识别，GPT-4o 基于结构化检测结果做问题表达。这样既能保证检测有稳定依据，也能控制截图数据外传风险，让最终报告真正服务设计师和开发人员。

# uidesign 项目协作规则

## 项目定位

`uidesign` 是一个面向设计师和前端团队的设计还原度走查小工具。

它的核心目标是：减少人工对比设计稿和开发页面的成本，帮助团队快速发现「开发出来的页面和设计稿不一致」的问题，并把问题汇总出来，方便后续修正。

这个项目关注的是设计还原度，不只是页面能不能打开，也不是只看大概像不像，而是尽量找出视觉、文字、间距、颜色、尺寸、位置等细节差异。

## 项目简介

项目要解决的问题是：开发后的页面还原度不高时，设计师或验收人员需要花很多时间手动对比设计稿和实际页面。

工具应该支持输入设计稿截图和开发页面截图，自动进行对比，识别差异，并输出清晰的问题汇总。

目前还原度走查的精度不够好，后续开发要重点提升检测准确性，尤其是：

- 页面结构差异
- 组件位置差异
- 字体、字号、字重差异
- 文本内容差异
- 颜色差异
- 间距、边距、对齐差异
- 图片、图标、按钮等元素差异
- 漏开发、多开发、错开发的问题

## 技术方向

当前优先考虑的技术组合是：

- OpenCV：用于图像对比、区域检测、像素差异、边缘检测、布局差异分析。
- PaddleOCR：用于识别截图中的文字内容，检查文本是否正确、是否缺失、是否位置异常。
- GPT-4o：用于理解设计稿和页面截图中的视觉差异，辅助生成更像人话的问题说明和走查报告。

技术实现上要尽量把「机器检测」和「大模型理解」结合起来：

- OpenCV 负责做稳定、可重复的视觉检测。
- PaddleOCR 负责把图片里的文字识别出来。
- GPT-4o 负责把检测结果解释成人能看懂的问题描述。

不要只依赖大模型肉眼判断，也不要只输出一堆难懂的算法结果。最终结果要服务设计师和开发人员，让他们知道哪里不一致、为什么不一致、应该改什么。

## 说明

以后回答用户时，要尽量用用户能听懂的话解释。

如果涉及专业名词，要先说清楚它的简单意思，再说明它在项目里有什么用。

例如：

- 不要只说「用 CV 做 diff」。
- 应该说「用 OpenCV 对两张图做视觉对比，找出颜色、位置、大小这些肉眼容易漏掉的差异」。

所有说明都要围绕实际问题展开：这个功能解决什么问题、为什么这么做、会带来什么效果。

## UI 风格

这个工具是给设计师和前端验收使用的，UI 风格应该专业、清晰、克制，不要做成花哨的营销页。

界面应该优先支持高效走查：

- 清楚展示设计稿、开发页面和差异区域。
- 差异标记要醒目，但不要影响用户看图。
- 问题列表要方便筛选、定位和复查。
- 颜色、间距、字体等信息要尽量结构化展示。
- 报告页面要适合导出和分享。

视觉风格建议：

- 干净的浅色背景为主。
- 使用清晰的层级和留白。
- 重点区域可以用对比色标记。
- 不要过度使用渐变、大面积装饰图形或复杂动画。
- 按钮、标签、状态提示要让设计师一眼看懂。

## 开发协作规则

开发时优先保证功能可靠、结果清楚、改动可控。

协作时遵守这些规则：

- 修改代码前，先看清楚现有项目结构和已有写法。
- 不要为了炫技引入不必要的复杂架构。
- 能沿用项目现有技术和风格时，优先沿用。
- 每次改动都尽量聚焦一个明确目标。
- 不要顺手重构无关代码。
- 不要随便改动用户没有要求修改的文件。
- 不要乱删文件。
- 如果确实需要删除、移动或大范围重构文件，必须先说明原因并征得用户同意。
- 回答用户时，要用直白的话说明做了什么、为什么这么做、怎么验证。
- 如果有风险或不确定的地方，要直接说清楚，不要装作已经确定。

涉及算法精度时，要说明：

- 当前方案能检测什么。
- 当前方案检测不了什么。
- 哪些结果可能误判。
- 后续可以怎样提高准确率。

## 安全与配置规则

项目里的密钥、Token、账号、接口地址等敏感配置不能写死在代码里。

安全和配置方面遵守这些规则：

- 不要把 API Key、访问令牌、账号密码提交到仓库。
- 敏感配置使用环境变量或本地配置文件管理。
- 本地配置文件如果包含密钥，应该加入 `.gitignore`。
- 给出配置示例时，只写占位符，不写真实密钥。
- 调用 GPT-4o、OCR 服务或其他外部服务时，要考虑失败、超时和费用问题。
- 上传设计稿或页面截图时，要注意图片可能包含未公开的产品信息。
- 处理用户文件时，不要擅自删除、覆盖或上传。
- 如果需要联网、调用外部 API、写入文件或批量处理数据，要先说明用途。

默认开发原则：先保证数据安全，再追求自动化效率。
